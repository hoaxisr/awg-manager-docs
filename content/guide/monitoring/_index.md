Автоматическая проверка связности через туннель и перезапуск при потере соединения. awg-manager периодически отправляет тестовые пакеты через интерфейс туннеля на заданную цель; при серии сбоев туннель рестартует.

Мониторинг настраивается **для каждого туннеля отдельно**. Механизм проверки различается в зависимости от backend-режима туннеля (см. [Быстрый старт — выбор backend](../../quickstart/#шаг-1--выбрать-режим-backend)).

## Открыть раздел

В главном меню — **Мониторинг**.

![Страница Мониторинг](/img/guide/monitoring/tab.png)

Страница показывает статус-карточки всех туннелей и журнал последних проверок. Кнопка **Проверить** вверху запускает разовую проверку для всех туннелей.

## Kernel-туннели

Для туннелей в режиме **Kernel** awg-manager использует **собственный механизм проверки** (не требует компонентов Keenetic OS).

![Настройки мониторинга для Kernel-туннеля](/img/guide/monitoring/kernel-settings.png)

Параметры:

| Параметр | Значение |
|---|---|
| **Метод** | `HTTP 204` (запрос пустого 204-ответа) или `ICMP` (ping) |
| **Цель** | IP или хост для проверки, например `8.8.8.8` |
| **Интервал** | 10–600 секунд между проверками в норме |
| **Интервал при dead** | 30–600 секунд при статусе `dead` (реже, чтобы не нагружать систему) |
| **Порог ошибок** | 1–20 подряд неудачных проверок перед признанием dead |

## NativeWG-туннели

Для туннелей в режиме **NativeWG** awg-manager делегирует мониторинг **встроенному механизму Keenetic OS** — компоненту `ping-check` (NDMS). На карточке туннеля в разделе Мониторинг это обозначено бейджем **NDMS**.

{{< callout type="warning" >}}
Требуется установленный компонент **ping-check** в прошивке Keenetic. Если он отсутствует, awg-manager покажет предупреждение в разделе Мониторинг. Установить — веб-интерфейс роутера → **Управление** → **Настройки системы** → **Изменить набор компонентов** → включить **ping-check**.
{{< /callout >}}

![Настройки мониторинга для NativeWG-туннеля](/img/guide/monitoring/nativewg-settings.png)

Сверху модалки — быстрые пресеты:

- `ICMP 8.8.8.8` / `ICMP 1.1.1.1` — ping Google / Cloudflare
- `TCP 8.8.8.8:53` — подключение к DNS-порту Google
- `TLS 1.1.1.1:443` — TLS-handshake к Cloudflare

Параметры:

| Параметр | Значение |
|---|---|
| **Хост** | IP или хост |
| **Метод** | `ICMP` (ping), `TCP Connect` (подключение к порту), `TLS` (TLS-handshake) |
| **Интервал (сек)** | 3–3600 — пауза между проверками |
| **Максимум сбоев** | 1–10 — неудач подряд до признания dead |
| **Минимум успехов** | 1–10 — удач подряд до возврата в healthy |
| **Таймаут (сек)** | 1–10 — ожидание ответа на пробу |
| **Перезапуск при dead** | Тумблер — автоматически перезапускать туннель при потере связи |

Пределы на числовые параметры заданы самим компонентом `ping-check` Keenetic NDMS — изменить границы нельзя.

`TCP Connect` и `TLS` проверяют не только ICMP-доступность, но и что через туннель можно установить полноценное соединение — полезно, если ICMP блокируется провайдером или сервером.

## Карточка статуса

На каждой карточке виден:

- **Статус туннеля** (running / dead / disabled) с цветным индикатором
- **Последняя проверка** — время, метод, результат
- **Счётчики** подряд удачных и неудачных попыток
- **Тумблер** включения/отключения мониторинга для этого туннеля
- **Шестерёнка** — открыть модалку настроек (соответствующую backend-режиму)

## Журнал проверок

Ниже статус-карточек — таблица **Журнал проверок** с колонками:

- **Время** — когда выполнена проверка
- **Туннель** — к какому туннелю относится
- **Результат** — `INIT` (инициализация), успех, сбой или перезапуск
- **Задержка** — RTT в миллисекундах
- **Счётчик** — текущее значение в формате `сбои/порог` (например `0/3`)

Сверху журнала — фильтр по туннелю, кнопки **Копировать** (в буфер обмена) и **Очистить** (стереть историю).

## Отключение мониторинга

Тумблер "Мониторинг" на карточке или в модалке настроек. При отключении: проверки прекращаются, журнал сохраняется. Включение обратно — сразу активно с сохранёнными настройками.

Для NativeWG-туннелей отключение снимает конфигурацию с NDMS pingcheck; для Kernel-туннелей останавливает внутренний поток проверки awg-manager.
